作業系統

intro


Operating System is a resource allocator and control programs

  • Resource allocator
    • CPU time,memory,storage,I/O devices,etc.
  • Control program
    • It needs to prevent errors and improper access to the computer

von Neumann Architecture

儲存程序電腦在體系架構上主要特點有：

  • 以運算單元為中心
  • 採用儲存程序原理
  • 記憶體是按位址訪問、線性編址的空間
  • 控制流由指令流產生
  • 指令由操作碼和位址碼組成
  • 資料以二進位編碼
    Main Memory -> Temporary storage(faster)
    Hard Disk -> Permanent storage(slower)
    Motherboard(主機板)
    透過南橋(southbridge chip)處理低速訊號、北橋(northbridge chip)處理高速訊號

Boot

  1. 透過小小的程式碼載入kernel
  2. kernel載入記憶體然後啟動
  3. kernel初始化硬體
  4. 根文件系统(root file system)掛載 <- 加载其它文件系统的”根“
  • MBR - Master Boot Record :用來引導Boot loader，MBR通常在硬碟的第一個扇區(Sector0)，大小大約512Bytes;主要分成兩個部分
  1. 引導加載程式(Boot loader)： MBR 中的前面一小部分用於存放引導加載程序，它是在電腦啟動時執行的程式碼。這段程式碼負責載入操作系統的引導扇區，從而啟動操作系統。
  2. 分區表(Partition Table)：MBR 中的剩餘部分包含分區表，記錄了硬碟上的分區信息。分區表可以容納最多四個主分區，或者三個主分區和一個擴展分區。擴展分區可以包含多個邏輯分區。每個分區表選項（Partition Entry）通常占用 16 位元組，其中包括分區的起始位置、大小和分區類型等資訊。

執行流程大概是這樣：

舊的電腦通常是BIOS+MBR
新的會用UEFI(Unified Extensible Firmware Interface)，UEFI比傳統硬碟多了以下功能：

  1. 圖形化介面： UEFI提供了一個圖形界面，相對於傳統BIOS的文字界面更加現代化。這使得使用者可以使用滑鼠和鍵盤進行交互，更容易進行設置和配置。
  2. 更大的硬碟支援： UEFI支援更大的硬碟容量，相較於BIOS的2TB限制，UEFI能夠處理更大容量的硬碟，這在現代存儲需求日益增長的情況下變得更為重要。
  3. 啟動安全性： UEFI引入了安全啟動（Secure Boot）功能，這是一種防範恶意軟體攻擊的機制。Secure Boot確保只有由已經數位簽名的合法引導載入程序才能啟動。
  4. 模組化設計： UEFI是模組化的，這意味著支援程式碼和驅動程式可以被動態加載，而不是像BIOS一樣需要將所有的驅動程式都集成到固件中。
  5. 擴展性： UEFI是可擴展的，允許廠商添加自己的功能模組。這使得硬體製造商和軟體開發者能夠更靈活地擴展和改進系統功能。

原本的MBR被GPT(GUID Partition Table)取代了，GPT 提供了更大的容量支援、更靈活的分割和識別方式，以及更強大的錯誤檢測和修復機制。這使得 GPT 成為 UEFI 系統中常見的磁碟分割表格式。在使用 UEFI 引導的系統中，通常會使用 GPT 來替代舊的 MBR 分割表。

The issues in designing an OS

  • The CPU idle time is excessive and it is hard to estimate the completion time of the user jobs

Reasons：

  1. The manual set-up time is too long
  2. The speed of I/O

Solutions：

  1. Resident monitor
  2. Introducting faster devices to mediate CPU and I/O speed
  3. Multiprogramming

2.Introducting faster devices to mediate CPU and I/O speed

  1. Off-line CPU(控制CPU離線在線)
    2.1 Buffering(緩衝區)
    2.2 Spooling

  • Benefits
    • 資源使用率上升
    • 如果操作是有效的，CPU閒置時間趨近於0
  • Drawbacks
    • 需要手動取消載入
    • 只支持有順序的access
      2.1
      記憶體中的一個空間，將 I/O 的資料先收集至 buffer 中。
      等待至 CPU 可運算的資料量時，CPU 再將資料由 buffer 中一次取出計算。
      在 CPU 開始計算的同時，I/O 則可繼續收集資料，達到 CPU 與 I/O 可同時運行的目的。

2.2
SPOOL
SPOOL 全名『Simultaneous Peripheral Operations On-Line』

是將磁碟機視為一個大 buffer，資料讀取與寫入時，不直接讀取，而由磁碟機中讀取。

例如有很多筆資料要交給印表機輸出時，可將待印資料的清單放在一個表格中，印表機印完一筆資料後，可直接至表格讀取下一比待印資料，而 CPU 把待印資料的清單放在表格中後，可以去做別的計算，I/O 也可繼續執行。

  • Buffering 與 SPOOL 的不同
    • SPOOL 可允許某個工作的 CPU 計算（Computation）與另一個工作的 I/O 操作（Operation）同步執行（Overlay execution）。
    • Buffering 只允許許某個工作的 CPU 計算與 I/O 操作同步執行。

3. Multiprogramming

多重程式(Multiprogramming)

  • 概念:
    • 當某程式暫時不需使用CPU的時候，監控程式會啟動另外的正在等待CPU資源的程式。目的是讓CPU資源能充分被使用。
  • 優點:
    • 看似原始但在當時確實大幅提高了CPU使用率。
  • 缺點:
    • 程式之間的排程策略太過粗糙，因為CPU是前一程式不使用時才能輪到其他需要CPU資源的程式使用CPU，也就是程式沒有優先序高低的差別；這會讓一些急需使用CPU完成任務的程式(例如使用者互動的任務)，可能需要等待很長的時間才分配到CPU資源。這會造成你在你的laptop按了一下滑鼠、或者在鍵盤上敲了幾個鍵，結果過了10分鐘系統才有反應。

多工(Multi-tasking)系統

因為前兩種排程方式的缺點，因而演進成我們現在很熟悉的多工(Multi-tasking)模式。

  • 概念: 使用一種叫先佔式(Preemptive)的CPU分配方式
    • 作業系統接管了所有硬體資源，且本身是執行在受硬體保護的級別。
    • 所有程式都以行程(Process)的方式執行在比作業系統權限更低的級別，每個行程都有自己獨立的位址空間、有單獨的記憶體，使行程間的位址空間互相隔離。
    • CPU由作業系統統一分配:
      • 每個行程根據行程優先序都有機會得到CPU資源，但如果執行時間超過一定時間，作業系統會暫停該行程，將CPU資源釋出給其他等待執行的行程。
      • 作業系統可強制剝奪CPU資源並分配給它認為目前最需要的行程。
      • 如果作業系統分配給每個行程的時間都很短，即CPU在多個行程間快速切換，看起來會像是很多行程都在同時執行。
    • 目前幾乎所有現代OS都是採用這種方式，比如UNIX、Linux、Windows NT以後的版本、Mac OS X以後的版本。

Concurrency VS Parallelism

  • Concurrency：並發性
  • Parallelism：並行性

用早餐店的工作解釋：早餐店有A,B共二位員工以及三種類型的工作內容（1.烤土司、2.煎蛋與火腿、3.組合1~2做成火腿蛋土司)

  • Concurrency：A在烤土司的同時會去煎蛋和火腿，而B會隨時注意土司烤好沒，同時把A煎好的蛋與火腿，一起組合成火腿蛋土司。
  • Parallelism：A、B分別做自己的火腿蛋土司，不共享食材。

computer-system organization

  • Interrupts:程式向系統要求服務的工具
    當硬體或軟體傳來優先事件信號時，立即停止正在執行的工作，轉向優先執行該事件，並把被中斷程式的位址傳到interrupt table，等kernel做完傳回資料，就不會忘記剛剛做到哪裡
  • Interrupts table:
    紀錄 interrupt 在哪個程式執行到哪(哪個記憶體片段)
    contains the memory address(es) of all the service routines (ISR)
    採用Stack方法將中斷時每一步驟之位址依序紀錄於特定堆疊中，等待中斷事件執行完畢後，在由堆疊中取得各位址，反序找到中斷位址，繼續執行原來之工作

現在的電腦都是 Interrupt，而 Interrupt 分為兩個。

I/O 間傳送的方法

  • Polling
    • 最簡單的方式讓 I/O 裝置與 CPU 溝通。
    • I/O 裝置只要將資訊放進狀態暫存器，CPU 會周期性的檢查並取得資訊來得知需要服務的裝置。
  • Interrupt-driven I/O
    • 利用 Interrupt 的機制，當一個 I/O 裝置需要服務時，會發出 Interrupt 來通知 CPU（非同步 I/O）。
  • DMA （Direct Memory Access）
    • 提供一個裝置控制器，讓 I/O 裝置能夠直接在記憶體做資料的傳輸，不需要 CPU 的參與。
  • Cycle Stealing
    • DMA 向 CPU 竊用機器週期，而直接向記憶體存取資料，因此 CPU 可與週邊裝置的工作同時進行。 -萬一 CPU 與 DMA 對記憶體存取發生衝突，則給 DMA 較高的優先權（執行消耗小的先執行）。

System Model

  • 每個資源類型都有實例(instances)
  • 每個process要資源都是:
    • request
    • use
    • release
  • 一組process相互等待對方擁有的資源（稱為循環等待(Circular Waiting)），導致所有process無法運行，進而降低了CPU利用率和生產力的情況被稱為死結。
  • 如果下面四個條件同時成立可能會造成死結:

    • 必要條件:

      • 互斥（Mutual Exclusion）： 同一時間只有一個線程或進程能夠使用一個資源，這是為了防止資源衝突。
      • 持有並等待（Hold and Wait）： 一個進程或線程持有至少一個資源，同時等待獲取其他被其他進程或線程持有的資源。這創造了進程間的相互依賴。
      • 不可剝奪（No Preemption）： 資源不能被強制性地從一個進程或線程中釋放，只能由持有資源的進程或線程願意釋放。
      • 循環等待（Circular Wait）： 存在一個進程等待序列，其中每個進程都在等待下一個進程所持有的資源，形成了循環。

    • 充分條件:

      • 單一實例資源的循環等待（Circular Wait with Single Instance in Every Resource）： 在每個資源類型中僅有一個實例，同時存在一個進程等待循環。

With Deadlock

  • 沒有circle -> 沒有deadlock

  • 有circle

    • 如果每個資源類型僅有一個實例，那麼可能發生死結（充分條件：單一實例資源的循環等待）
    • 如果每個資源類型有多個實例，則有死結的可能性

  • 為了防止死結，您可以使四個必要條件中的其中一個失效。讓我們探討每個條件以及如何使其失效：

    • 獨佔（Mutual Exclusion）：對於可共享資源（例如，只讀文件）不是必要的；對於不可共享資源必須滿足，使獨佔失效非常困難!!因為獨佔是某些資源固有的屬性，沒有辦法打破它！
    • 持有並等待（Hold and Wait）：必須保證每當一個線程請求資源時，它並不持有任何其他資源
      • 資源請求的問題被劃分為兩種方法
        1. 資源利用率低
        2. 可能發生飢餓
    • 不可剝奪（No Preemption）：如果一個進程持有某些資源，並請求另一個不能立即分配給它的資源，則必須釋放目前持有的所有資源。
      • 使不可剝奪失效，允許進程從其他等待的進程中抢占資源。（即，使其具有抢占性）。這可能會導致飢餓，但不會有死結。（可以使用“老化”技術解決）然而，像印表機這樣的資源不太適用於這種方法。
    • 循環等待（Circular Wait）：強制對所有資源類型進行總排序，並要求每個線程按照列舉的遞增順序請求資源。

  • 總結：

    • 優勢：確保系統永遠不會發生死結。
    • 缺點：資源利用率低，吞吐量低。

Deadlock Avoidance

  • 單一實例的資源類型（Single instance of a resource type）：
    • 對於單一實例的資源，可以使用「資源分配圖」（Resource-Allocation Graph）來表示和管理資源的分配狀態。這種圖形表示法能夠展示進程和資源之間的關係，以及資源的分配和釋放。通過觀察這個圖，系統可以進行死結的檢測和避免。
  • 多個實例的資源類型（Multiple instances of a resource type）：
    • 對於多個實例的資源，可以使用「銀行家算法」（Banker’s Algorithm）來進行資源的分配和死結避免。銀行家算法包括一個「安全算法」，用於確保系統在任何時刻都處於安全狀態，從而避免死結。這種方法需要在系統中事先知道每個執行緒可能需要的最大資源量，以便動態地進行資源分配。

單一實例的資源類型（Single instance of a resource type）：

  • 聲明邊（Claim Edge，Ti → Rj）：
    • 代表進程 Tj 声明在未來可能會請求資源 Rj。
    • 以虛線表示。
  • 請求邊（Request Edge，Ti → Rj）：
    • 當一個執行緒（Ti）實際請求資源（Rj）時，由聲明邊轉換而來。
    • 這種轉換通常由作業系統進行。
  • 指派邊（Assignment Edge，Ti → Rj）：
    • 當所請求的資源（Rj）被分配給執行緒（Ti）時，由請求邊轉換而來。
    • 表示該執行緒目前擁有該資源。
  • 釋放資源：
    • 當一個執行緒釋放資源時，指派邊重新轉換為聲明邊。
    • 這表示執行緒可能在未來再次請求該資源。
  • 事先聲明需求：
    • 系統中的資源必須在事先聲明，即每個執行緒必須事先聲明其可能的資源需求。
    • 這些信息用於在資源分配圖中建立聲明邊。

      不安全的邊OS不會允許，OS需要N^2去檢查(N代表有N個processs/threads)

多個實例的資源類型（Multiple instances of a resource type）

銀行家演算法(Banker’s Algorithm)：

  • 複雜度O(m x n^2)
    • n = processes，m = resource types
  • 事先聲明最大使用量：
    • 每個執行緒必須在系統開始運行之前事先聲明它可能需要的每種資源的最大使用量。
  • 資源請求可能需等待：
    • 當一個執行緒請求資源時，可能需要等待，直到資源可用。這是因為系統需要確保資源的分配不會導致死結。
  • 執行緒獲得全部資源後必須歸還：
    • 當一個執行緒獲得了它所需的全部資源，它必須在有限的時間內歸還這些資源。這有助於防止執行緒獲得資源後無限期地持有它們，從而影響其他執行緒的執行。
  • 判斷系統是否會進入死結：
    • 銀行家算法包括一個「安全算法」，用於動態地評估系統是否處於安全狀態。安全算法確保在執行過程中的任何時刻，系統都能夠分配資源而不會進入死結。

CPU 排程算法

  1. 先來先服務（FCFS）排程：
    • 最先到達的進程會先被執行。
    • 簡單易懂，但可能會導致”僵直效應”，即較短的進程被長進程擋住。
  2. 最短工作優先（SJF）排程：
    • 具有最短執行時間的進程會優先被選擇。
    • 可降低等待時間和提高周轉時間，但需要提前知道進程的執行時間，這通常是不可能的。
  3. 最短剩餘時間優先（SRTF）排程：
    • 也稱為搶占式 SJF，允許具有更短剩餘執行時間的進程抢占當前正在執行的進程。
    • 可以減少等待時間，但同樣需要提前知道執行時間。
    • TAT = CT-AT
    • WT = TAT-BT
  4. 輪轉（Round-Robin，RR）排程：
    • 每個進程被分配一個時間片，當時間片用盡時，該進程會被暫停，然後換到下一個進程。
    • 簡單且公平，但可能導致長等待時間。
  5. 優先權（Priority）排程：
    • 進程根據優先級被選擇執行，優先級越高，越早執行。
    • 可以根據任務的緊急性或重要性調整優先級。
  6. 多層次佇列（Multilevel Queue）排程：
    • 進程被劃分為不同的優先級，每個優先級都有自己的佇列，並且這些佇列之間可能有不同的排程算法。
  7. 多層次反饋佇列（Multilevel Feedback Queue）排程：
    • 類似多層次佇列，但這裡進程可以在不同的佇列之間移動，根據其執行行為進行調整。

等待時間（Waiting Time）= 周轉時間(Turnaround Time) - 處理時間(Brust Time)
周轉時間（Turnaround Time） = 完成時間(complete Time) - 執行時間(Arrival Time)
